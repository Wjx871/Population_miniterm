## 2. 项目定位与目标

项目为“人口数据库管理系统”，围绕“户 - 人 - 证 - 迁移记录”一体化管理，采用前后端分离架构：

- 前端：Vue 3 + Vite + Element Plus + Pinia + Vue Router + Axios + ECharts；
- 后端：Spring Boot + MyBatis/MyBatis-Plus 风格的 REST API；
- 数据库：MySQL，默认库名 `population_miniterm`；
- 本地服务约定：后端 `http://localhost:8080`，Vite 开发端口偏好 `5180`，`/api` 代理到后端。

最小演示闭环为：

```text
登录
  -> 按三级权限展示菜单与按钮
  -> 人口档案查询/维护
  -> 家庭户与成员维护
  -> 迁入或迁出申请
  -> 查看我的申请
  -> 超级管理员审批
  -> 查看结果、当前户籍与历史归档
```

## 4. 不可偏离的前端规则

- 固定三级角色：`NORMAL_USER`（普通用户）、`HOUSEHOLD_ADMIN`（户口管理员）、`SUPER_ADMIN`（超级管理员）。角色名称/代码以最终后端定义为准，前端不可另造一套。
- 普通用户以查询查看为主；户口管理员办理常规业务、提交重大业务申请；超级管理员拥有完整菜单、系统管理和审批权限。
- 未登录必须跳登录；无权限菜单/按钮不可见；手动进入无权限路由必须进入 403；后端仍是最终权限边界。
- 列表默认脱敏身份证号、手机号、详细地址等敏感数据；详情按授权展示。敏感材料和导出文件不得通过公开 URL 暴露。
- 当前户籍与历史归档严格分离；人口页不得直接修改迁出/死亡/注销等重大状态；家庭户页不得直接销户。
- 列表页统一为“标题说明 - 查询区 - 操作栏 - 表格 - 分页 - 抽屉/弹窗”；复杂业务使用独立页或步骤条；关闭未保存表单需确认。
- 统一状态颜色：绿色（正常/有效/通过/办结）、蓝色（草稿/待处理/审批中）、橙色（即将到期/待补充/退回）、红色（驳回/失败/过期/异常）、灰色（停用/注销/撤回/归档）。
- 公共组件应优先沉淀复用：`PageContainer`、`SearchForm`、`CrudTable`、`StatusTag`、选择器、`SensitiveText`、`MaterialUploader`、`ApprovalTimeline`、`DetailDrawer`、`PermissionButton`、`EmptyState` 等；同类功能不得各写一份。
- 所有请求最终统一经 `src/utils/request.ts`（当前实际路径为 `frontend/src/api/request.js`，后续重构须统一，不要新增第三套实例）。

### 已知工程与接口风险

- `frontend/src/api/person.js`、`statistics.js` 自建 Axios 实例；`persons.js` 与 `person.js` 存在重复人口 API。这违反“单一请求实例”目标，应在 M0 统一后再删除冗余文件（删除前必须确认实际引用）。
- `request.js` 仅显式处理 401；尚无 403 跳转、重复错误抑制、下载、请求取消等统一能力。
- Store 当前只保存 `roleName`，没有权限等级、权限码或菜单/按钮权限计算；路由守卫只做登录判断。
- 主布局菜单是固定平铺项，所有已列模块均可见；与三级权限和分组菜单要求不符。
- 当前代码中存在 `auth/register`、`households`、`migrations`、`certificates`、`users` 等前端调用，但当前后端源码只确认有 `/api/auth/login`、`/api/persons`、`/api/residents`、`/api/statistics`、`/api/system/health` 控制器。接口字段、分页格式和真实可用性必须先联调确认，不能仅凭前端 API 文件假设已实现。
- `dashboard.js` 调用的 `/persons/statistics`、`/migrations/in/statistics`、`/certificates/expire-soon` 目前未在已扫描后端 Controller 中确认；数据看板尚未接入 ECharts。
- 多个当前页面直接提供删除、迁入/迁出办理入口；后续重构必须收敛为权限控制与申请审批模型，避免前端直接办结重大操作。
