# 后端 V1 范围冻结

基线与状态来源见 `backend-completion-audit.md`。V1 的目标是课程项目“可运行、可演示、数据可追溯、权限可解释”，不追求生产政务平台规模。

## 1. V1 必须完成

| 分组/能力 | V1 | 当前状态 | 完成标准 | 不完成影响 | 阻塞其他模块 |
|---|---|---|---|---|---|
| 统一人口主模型 | 是 | DUPLICATE | 明确 `person` 为唯一主模型；旧 residents API 退役/兼容策略明确；核心人口无物理删除 | 数据口径、外键和统计持续分裂 | 是 |
| 身份证与手机号校验 | 是 | MISSING/PARTIAL | 身份证校验位、日期、生日/性别一致性；手机号规则统一；单元测试覆盖边界 | 脏数据进入所有业务 | 是 |
| 家庭户档案 | 是 | MISSING | 分页、详情、新增、修改、状态管理，Mapper/XML/权限/范围/审计齐全 | 无法完成“先立户”主流程 | 是 |
| 家庭成员与户主关系 | 是 | MISSING | 查询、加入、移出、安全变更户主；唯一有效归属和户主必须为有效成员 | 迁移/注销只能依赖预置数据 | 是 |
| 通用证件 | 是 | MISSING | 登记、查询、修改、状态变更；编号唯一、有效期校验、脱敏和审计 | “户—人—证”模型不闭环 | 否 |
| 部门基础查询 | 是 | PARTIAL | 至少提供受权只读查询；管理 CRUD 可限管理员 | 用户/范围配置不可演示 | 否 |
| 行政区划基础查询 | 是 | MISSING | 区划表、层级/代码查询 API、索引和演示数据 | 地址与数据范围只能写自由码 | 是 |
| 数据字典基础查询 | 是 | PARTIAL | MySQL/H2 同构，按类型查询；写接口仅管理员或不纳入 V1 | 状态值无法统一展示 | 否 |
| 当前户籍与历史 | 是 | PARTIAL/COMPLETE | 当前登记可查；历史归档保留；不物理删核心主档 | 户籍详情不闭环 | 是 |
| 迁入迁出与同市迁移 | 是 | COMPLETE/PARTIAL | 复用现有审批+显式执行；补同市跨区契约和测试，不重写 | 核心状态流转不完整 | 否 |
| 人口/家庭户注销 | 是 | COMPLETE | 保持现实现和归档规则 | 主档生命周期不完整 | 否 |
| 流动人口与居住证 | 是 | COMPLETE | 保持登记、关闭、申请、签发、签注、注销闭环 | 重要业务缺失 | 否 |
| 重点人口 | 是 | MISSING | 建档、解除、历史/跟踪，敏感权限、数据范围、审计、显式执行；表与迁移齐全 | 需求 P1 和数据库设计缺口 | 否 |
| 统一申请/材料/审批/执行 | 是 | COMPLETE | 单级审批；批准后显式 execute；不得增加 ApprovalGate 或第二状态机 | 专业业务规则分裂 | 是 |
| 综合人口/家庭户/迁移查询 | 是 | MISSING/PARTIAL | 按人、户、证、状态、区划、时间组合查询；统一范围和脱敏 | 无法展示一体化模型 | 否 |
| 首页与大屏统计 | 是 | PARTIAL | 基于冻结主模型和业务表定义口径；覆盖人口、迁移、流动、重点、证件 | 展示数据失真 | 否 |
| 普通/敏感导出与下载审计 | 是 | COMPLETE | 保持权限分离和审批执行 | 敏感数据泄露风险 | 否 |
| 操作/登录日志 | 是 | PARTIAL/MISSING | 重大写操作、登录成功/失败均可查；范围、脱敏、保留策略明确 | 无法追责 | 否 |
| MySQL/H2/迁移/演示数据 | 是 | PARTIAL | 主脚本初始化；迁移从基线升级且可重复；H2 覆盖生产表；真实 MySQL 全回归 | 无法交付部署 | 是 |
| API 与部署文档 | 是 | PARTIAL | 所有 V1 API、权限、状态、错误和演示步骤可查 | 无法稳定验收 | 否 |

## 2. V1 不包含

- 多级审批、工作流编排平台、消息队列、微服务、云存储。
- 真实身份证联网核验、真实制卡、政务系统联网、短信、邮件。
- Redis 权限缓存、分布式缓存、分布式事务、高可用集群。
- 大规模异步报表、500 并发/百万级数据的生产级性能承诺、真实灾备平台。
- 分户/并户若需求方未将其列为最终演示用例，可作为 V1.1；V1 只保证立户、成员、户主变更和销户闭环。

## 3. 架构冻结

- 保持 Java 17、Spring Boot 3.5.3、普通 MyBatis Mapper+XML、Spring Security、`@PreAuthorize`、`ApiResponse<T>`、`CurrentUserContext`、`DataScopeCriteria`、MySQL 8、H2。
- 保持单级审批和“批准后显式 execute”。
- 不引入 MyBatis-Plus、第二套 JWT、Redis 权限缓存、`Result<T>`、`ApprovalGate` 或第二套状态机。
- 不物理删除 `person`、`household` 等核心主档。
- develop 为唯一事实来源；不得合并 `origin/household-migration`，不得以 unrelated histories 引入另一套后端。

## 4. 变更控制

新增需求必须先判断是否支持上述 V1 主流程。若不影响“户—人—证—迁移—审批—查询—审计”闭环，默认进入 V1.1。任何数据库变化必须在同一阶段同时更新 MySQL 主脚本、增量迁移、H2 schema、演示数据、Mapper/XML、测试和 API 文档。
