# 后端完成路线图

路线按依赖排列：先统一人口主模型和校验，再完成家庭户主数据，再完成证件/重点人口，之后才做综合查询与统计，最后做横切安全和真实 MySQL 封版。现有迁入迁出、审批、注销、流动人口、居住证和导出只做回归，不重复开发。

## Phase 08：人口主模型与家庭户主数据收口

1. **目标**：以 `person` 为唯一人口主模型，完成严格身份校验、家庭户、成员与户主变更闭环。
2. **现在做的原因**：所有证件、重点人口、综合查询和统计都依赖稳定的人—户关系。
3. **前置依赖**：`develop@8e6136d`；现有认证、权限、数据范围基础；阶段 01—07 测试通过。
4. **具体功能**：双模型处置；人口软删除规则；家庭户分页/详情/新增/修改；成员查询/加入/移出；安全户主变更；当前户籍查询；严格身份证/手机号校验。
5. **数据库变化**：新增 V4_007，补唯一有效成员、户主/成员一致性所需索引与约束；制定 residents 数据退役/迁移策略；不得直接丢弃用户数据。
6. **API**：`/api/households` CRUD（不含物理删除）、`/{id}/members`、成员加入/移出、`/{id}/head`、当前户籍查询；Person API 统一。
7. **权限**：`household:view/create/update/member:manage/head:change`；沿用 `population:*`，不复用管理员万能权限。
8. **数据范围**：所有列表、详情和写入按 `DataScopeCriteria`；户、成员、人口的 region 必须一致。
9. **脱敏**：列表/详情中的身份证、电话、详细地址默认脱敏，完整查看沿用 `sensitive-data:view-full`。
10. **审计**：立户、修改、成员加入/移出、户主变更、旧模型处置均记录对象和结果。
11. **自动化测试**：身份算法、CRUD、唯一归属、户主合法性、越权/越范围、并发版本、事务回滚、无物理删除。
12. **MySQL 测试**：空库初始化、V4_007 从现基线升级、重复执行、约束冲突、演示数据、迁移/注销回归。
13. **完成标准**：Phase 08 任务书全部勾选；147 个既有测试不回退；新增测试通过；无双写和重复人口 API。
14. **风险**：residents 中可能存在 person 未覆盖的数据；家庭户约束会暴露旧演示数据不一致。
15. **回滚**：先做可逆数据复制/兼容读取；迁移提供反向 SQL；代码按独立提交回退，不删除旧表数据。
16. **建议分支**：`feat/backend-phase-08-household-master-data`。
17. **建议提交**：数据库与模型策略；身份证校验；家庭户查询；成员/户主写流程；测试和文档，分 5—7 个提交。
18. **并行性**：身份证算法可与家庭户只读查询并行；数据库约束、成员写流程和旧模型处置不可并行落地。
19. **预计工作量**：4—6 人日。
20. **下一阶段条件**：person 唯一事实源、户/成员接口可用、真实 MySQL 回归通过。

## Phase 09：证件、区划与字典基础能力

1. **目标**：补齐“户—人—证”及规范化基础数据。
2. **现在做的原因**：稳定的人户键和区域归属已由 Phase 08 提供；重点人口和综合查询需要这些基础维度。
3. **前置依赖**：Phase 08 完成。
4. **具体功能**：通用证件登记/查询/修改/状态变更；行政区划层级查询；部门只读查询；字典按类型查询。
5. **数据库变化**：V4_008 新增行政区划表/索引/种子；补 certificate 联合唯一与日期约束；H2 同步 data_dictionary。
6. **API**：`/api/persons/{id}/certificates`、`/api/certificates/{id}`、`/api/regions`、`/api/dictionaries/{type}`、`/api/departments/options`。
7. **权限**：证件查看/维护/状态权限分离；基础查询需认证；字典写入若非验收必需不开放。
8. **数据范围**：证件继承 person 范围；区划查询本身只读，业务使用时校验用户 region。
9. **脱敏**：证件号默认脱敏。
10. **审计**：证件新增、修改、状态变更；基础数据写入若开放必须审计。
11. **自动化测试**：证件唯一性/有效期/状态；区划层级；字典 H2 同构；越范围。
12. **MySQL 测试**：初始化、升级、重复执行、中文排序规则和索引检查。
13. **完成标准**：证件 CRUD 与三类基础查询可演示且文档齐全。
14. **风险**：当前居住证是专业表，不能与通用 certificate 双写；需定义引用/投影关系。
15. **回滚**：新增表可独立回滚；certificate 约束上线前先清理冲突并保留报告。
16. **建议分支**：`feat/backend-phase-09-certificate-master-data`。
17. **建议提交**：区划/字典 schema；基础查询；证件模块；安全与测试；文档。
18. **并行性**：区划/字典查询可与证件模块并行。
19. **预计工作量**：3—4 人日。
20. **下一阶段条件**：证件、区划、字典均可被业务稳定引用。

## Phase 10：重点人口专业业务

1. **目标**：完成重点人口建档、解除和历史/跟踪闭环。
2. **现在做的原因**：这是需求 P1 明确模块，依赖统一 person、区划、字典和审批。
3. **前置依赖**：Phase 08—09；现有单级审批、材料和显式执行。
4. **具体功能**：建档申请、详情/检索、材料、审批、显式执行、解除申请/执行、跟踪历史。
5. **数据库变化**：V4_009 完善 key_population，新增 followup/history 表、唯一有效档案和检索索引；H2 同步。
6. **API**：`/api/key-populations/applications`、查询/详情、`/{applicationId}/execute`、解除申请/执行、历史查询。
7. **权限**：view/apply/handle/execute/followup/view-sensitive 分离。
8. **数据范围**：按责任部门与 region；SELF 不得查看他人重点人口。
9. **脱敏**：默认最高等级脱敏；列表不得返回完整身份证/地址/原因详情。
10. **审计**：查询敏感详情、建档、解除、跟踪、导出均审计。
11. **自动化测试**：重复建档、非法解除、审批前执行、越范围、脱敏、回滚、并发。
12. **MySQL 测试**：V4_009 初始化/升级/重复执行、唯一约束、执行事务回滚。
13. **完成标准**：单级审批后显式执行，完整历史可查，不引入新状态机。
14. **风险**：敏感等级高；需求文档的提醒频率可只记录下次跟踪日，不引入消息队列。
15. **回滚**：新模块独立表；失败时回退代码和迁移，保留审计数据备份。
16. **建议分支**：`feat/backend-phase-10-key-population`。
17. **建议提交**：schema/domain；申请；审批监听/执行；查询安全；测试文档。
18. **并行性**：数据库/状态设计冻结后，查询与申请接口可部分并行；执行流程最后合并。
19. **预计工作量**：4—5 人日。
20. **下一阶段条件**：重点人口和所有基础维度均稳定，统计口径可冻结。

## Phase 11：综合查询、日志与统计口径

1. **目标**：形成可演示的一体化查询与可信统计。
2. **现在做的原因**：只有在主表和专业模块稳定后，查询 SQL 和统计口径才不会反复重写。
3. **前置依赖**：Phase 08—10。
4. **具体功能**：综合人口、家庭户详情聚合、迁移历史统一查询；登录日志查询；重大操作日志检索；首页/大屏统计补齐重点人口和证件口径。
5. **数据库变化**：仅增加经 EXPLAIN 证明需要的组合索引；不建第二套汇总库。
6. **API**：`/api/search/persons`、`/api/search/households`、`/api/search/migrations`、统计与日志接口完善。
7. **权限**：search/stats/log 权限分离。
8. **数据范围**：所有聚合从主查询层统一注入范围，禁止先查全量再内存过滤。
9. **脱敏**：统一 masking service；导出继续走现有 ExportWorkflow。
10. **审计**：敏感详情查询、日志查看和导出记录。
11. **自动化测试**：组合条件、分页稳定性、统计口径、越范围、脱敏、SQL 映射。
12. **MySQL 测试**：代表性数据集、EXPLAIN、结果对账和分页性能。
13. **完成标准**：相同口径在查询、统计、导出一致；无重复 Dashboard/Statistics 实现。
14. **风险**：复杂 SQL 容易漏范围或重复计数。
15. **回滚**：新增查询/API 可独立回退；索引使用 DROP INDEX 回滚。
16. **建议分支**：`feat/backend-phase-11-query-statistics`。
17. **建议提交**：查询契约；综合查询；日志；统计；索引与测试；文档。
18. **并行性**：日志可与综合查询并行；统计必须等待表和口径冻结。
19. **预计工作量**：4—5 人日。
20. **下一阶段条件**：核心功能均达到 DoD，统计对账通过。

## Phase 12：跨模块安全、一致性、MySQL 全回归与封版

1. **目标**：消除横切遗漏并形成可部署后端版本。
2. **现在做的原因**：封版检查必须覆盖完整 V1，而不是在模块开发中零散判断。
3. **前置依赖**：Phase 08—11 完成。
4. **具体功能**：Controller 权限矩阵、数据范围矩阵、脱敏/审计矩阵、异常码一致性、废弃 API 清理、README 对齐。
5. **数据库变化**：只修复回归发现的一致性问题；冻结 schema 版本。
6. **API**：生成最终清单，标记废弃端点，不新增业务范围。
7. **权限**：逐路由未认证/无权限测试。
8. **数据范围**：ALL/DEPARTMENT/REGION/SELF 的列表、详情、写入、导出全覆盖。
9. **脱敏**：响应、日志、异常、导出、文件名全检查。
10. **审计**：所有重大写操作和敏感读操作矩阵闭环。
11. **自动化测试**：全量 H2、静态重复路由/无权限检查、端到端回归。
12. **MySQL 测试**：空库初始化、全迁移、迁移二次执行、演示流程、事务回滚、约束和备份恢复演练。
13. **完成标准**：DoD 全部通过，文档与代码一致，标签前工作区干净。
14. **风险**：横切修复可能影响多个模块；必须小提交并持续回归。
15. **回滚**：每类安全修复独立提交；数据库修复提供前后校验和反向方案。
16. **建议分支**：`release/backend-v1-acceptance`。
17. **建议提交**：权限；范围；脱敏审计；MySQL；文档封版，分别提交。
18. **并行性**：矩阵审计可并行，修复合入需串行全回归。
19. **预计工作量**：3—4 人日。
20. **下一阶段条件**：全部验收通过后合并 develop 并创建后端 V1 标签。

## 阶段合理性检查

| 阶段 | 是否必要 | 是否重复开发 | 依赖是否满足 | 顺序是否合理 | 范围是否过大 | 可否拆分 | 风险 | 结论 |
|---|---|---|---|---|---|---|---|---|
| 08 主模型与家庭户 | 是 | 否 | 当前认证/流程基础满足 | 是，先稳基础表 | 中等 | 可按校验/查询/写流程拆提交 | 双模型数据迁移 | 第一优先级 |
| 09 证件与基础数据 | 是 | 否 | 依赖 08 | 是 | 适中 | 区划字典与证件可并行 | 居住证与通用证件边界 | 08 后执行 |
| 10 重点人口 | 是 | 否 | 依赖 08/09 和已有审批 | 是 | 适中 | 查询与流程可拆 | 高敏数据 | 不提前 |
| 11 查询日志统计 | 是 | 否 | 依赖 08—10 | 是，未在基础表不稳时做统计 | 中等 | 日志/查询/统计拆提交 | 漏范围、重复计数 | 后置合理 |
| 12 安全与封版 | 是 | 否 | 依赖全部功能 | 是 | 中等 | 按横切维度拆 | 跨模块回归 | 最终门禁 |

核验结论：不重复开发现有迁入迁出和审批；不引入第二状态机；高级能力未误列入 V1；每阶段可在数据库、Java、测试、文档内闭环并独立回归。原建议把重点人口放在家庭户之后正确，但区划/字典应在重点人口之前完成，因此调整为五阶段如上。
