# 第三阶段迁移事务设计

审批与执行刻意分离：审批确认材料与业务条件，形成 `APPROVED` 授权；经办人再次确认后执行，避免审批瞬间产生不可逆户籍变化。

通用申请保存流程与审计字段，`migration_in/out` 保存专业字段，`residence` 是当前事实，`residence_archive` 是历史事实。迁出快照包含姓名、身份证号、户号、地址、区划、登记类型及起止日期，所以后续修改人员或家庭户不会改写历史。

## 事务边界

迁入单事务：锁定申请/专业记录并校验版本，检查人员唯一当前户籍、目标家庭户和材料，校验跨区批次，插入 `residence`，创建或恢复 `household_member`，更新人员汇总状态，将专业记录和申请一起置为 `COMPLETED`，最后写审批轨迹与操作日志。

迁出单事务：锁定申请、专业记录、当前户籍和家庭户，校验成员/户主规则与材料，插入完整归档快照，删除当前户籍，将成员置为 `LEFT`，必要时变更户主或把家庭户置为 `PENDING_CANCELLATION`，更新人员汇总状态和两个完成状态，写入日志。任一步异常全部回滚。

幂等由状态条件更新、乐观锁、`UNIQUE(application_id)`、`UNIQUE(residence.person_id)`、唯一迁移批次和行锁共同保证。测试替换 `MigrationTransactionHook`，分别可在归档后、删除当前户籍后、创建当前户籍后注入异常；生产实现为空操作且不暴露制造故障的 HTTP 参数。

户主迁出且还有成员时，新户主必须是本户其他 `ACTIVE` 成员；最后成员迁出只进入待销户。本阶段不写 `CANCELLED`、不删除人员。第四阶段可基于 `PENDING_CANCELLATION` 新建独立销户申请与执行事务。

真实 MySQL 环境当前不可用：V4_003 已做静态检查，自动化使用 H2；仍需在可用 MySQL 8 实例上验证旧数据回填、约束建立和重复执行脚本。
