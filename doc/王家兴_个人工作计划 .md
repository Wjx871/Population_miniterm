### 2.3 业务申请与审批流程

业务申请和审批是本次数据库优化后的重点，也是系统区别于普通 CRUD 项目的核心亮点。

#### 统一业务链路

```text
业务申请
→ 材料上传
→ 材料核验
→ 提交审批
→ 审批通过或驳回
→ 执行业务操作
→ 记录日志
→ 必要时生成历史归档
```

#### 我的具体任务

- 业务申请创建；
- 草稿保存；
- 提交申请；
- 申请撤回；
- 材料列表关联；
- 审批任务生成；
- 审批通过；
- 审批驳回；
- 审批意见记录；
- 审批日志查询；
- 待办审批列表；

- 审批状态与业务状态联动。

#### 优先接入审批的业务

1. 人员销户；
2. 家庭户销户；
3. 重大迁出业务；
4. 重点人口登记与解除；
5. 高敏感人口数据导出。

### 2.4 迁入迁出和户籍归档事务

迁入迁出由赵书亚负责业务数据层和基础接口，我负责复杂流程、审批接入和事务一致性。

#### 迁入业务

迁入分为：

```text
同市跨区迁入
外来迁入
```

迁入成功后需要在同一事务中完成：

1. 写入迁入记录；
2. 建立或更新当前户籍登记；
3. 关联目标家庭户；
4. 更新家庭成员关系；
5. 更新人口当前状态；
6. 更新申请状态；
7. 写入操作日志。

#### 迁出业务

迁出分为：

```text
同市跨区迁出
迁往市外
```

迁出成功后需要在同一事务中完成：

1. 校验人员当前户籍；
2. 判断是否为户主；
3. 处理户主变更或家庭户状态；
4. 写入迁出记录；
5. 将当前户籍复制到历史归档表；
6. 移除当前有效户籍记录；
7. 更新家庭成员关系；
8. 更新人口状态；
9. 更新申请和审批状态；
10. 写入操作日志。

#### 关键要求

- 迁入、迁出和归档必须使用事务；
- 中间任意步骤失败时整体回滚；
- 不允许只更新状态而不保存历史记录；
- 不允许物理删除人口基础档案；
- 当前户籍与历史归档必须保持边界清晰。

### 2.5 销户业务

销户属于重大业务，必须经过审批，不能由普通经办人员直接完成。

#### 人员销户流程

```text
经办人员提交销户申请
→ 上传证明材料
→ 审批人员审核
→ 审批通过
→ 生成注销记录
→ 归档当前户籍
→ 移除当前户籍关系
→ 更新人口状态
→ 记录操作日志
```

#### 家庭户销户流程

办理前必须检查：

- 是否仍有有效家庭成员；
- 是否存在未完成迁移业务；
- 是否已完成户主和成员迁出；
- 是否存在未办结申请；
- 是否已上传必要材料。

#### 我的具体任务

- 销户申请接口；
- 销户前置校验；
- 审批通过后的业务执行；
- 注销记录写入；
- 户籍归档；
- 状态联动；
- 失败回滚；
- 操作日志记录。

### 2.6 操作日志与导出审计

#### 操作日志

需要记录：

- 操作用户；
- 操作模块；
- 操作类型；
- 操作对象；
- 操作时间；
- 操作结果；
- 请求路径；
- 脱敏后的修改前数据；
- 脱敏后的修改后数据。

重点记录：

```text
新增
修改
删除或停用
迁入
迁出
销户
审批
导出
登录
```

#### 导出审计

普通统计导出和敏感人口数据导出应区分处理。

高敏导出需要：

1. 校验权限；
2. 必要时提交审批；
3. 记录导出人；
4. 记录导出时间；
5. 记录查询条件；
6. 记录数据行数；
7. 记录文件信息；
8. 对身份证号和手机号进行脱敏。

## 三、项目统一规范

### 3.1 接口规范

所有接口统一采用：

```text
/api/模块名/资源名
```

示例：

```text
/api/auth/login
/api/users
/api/applications
/api/approvals/pending
/api/migrations/in
/api/migrations/out
/api/cancellations
```

统一返回格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

分页返回格式：

```json
{
  "records": [],
  "total": 0,
  "pageNum": 1,
  "pageSize": 10
}
```

### 3.2 状态码和异常规范

建议统一异常类型：

```text
参数校验异常
登录认证异常
权限不足异常
数据不存在异常
重复数据异常
业务状态异常
审批状态异常
数据库操作异常
```

常见业务提示应明确，例如：

```text
该人员已存在有效户籍登记
该人员已办理迁出，不可重复迁出
户主迁出前需要先完成户主变更
该申请当前状态不允许撤回
材料未核验完成，暂不能提交审批
当前用户无权查看该行政区划数据
```

### 3.3 Git 协作规范

建议分支：

```text
main
develop
feat/system-approval
feat/household-migration
feat/frontend-population
```

我的分支：

```text
feat/system-approval
```

提交信息示例：

```text
feat: add role and permission control
feat: implement approval workflow
feat: add migration transaction service
feat: implement cancellation approval
fix: rollback archive transaction on failure
docs: update API integration guide
```

#### 合并要求

- 功能完成后先自测；
- 通过编译和基础测试；
- 合并到 `develop`；
- 联调通过后再进入 `main`；
- 数据库字段变更必须先与赵书亚确认；
- 接口字段变更必须提前通知张锦源。

## 七、风险与应对措施

| 风险 | 可能影响 | 应对方式 |
|---|---|---|
| 数据库再次大改 | 后端实体和接口返工 | 教师确认后冻结核心表，后续仅做小调整 |
| 权限设计过于复杂 | 开发周期不足 | 先完成角色、等级和接口权限，数据范围做核心场景 |
| 审批流程功能过多 | 无法按时完成 | 优先完成提交、通过、驳回、撤回四个状态 |
| 迁出归档事务出错 | 数据不一致 | 统一使用事务并设计失败回滚测试 |
| 前后端字段不一致 | 联调延误 | 提前给出 DTO、VO 和接口文档 |
| 三人同时修改公共代码 | Git 冲突 | 公共代码由我审核，功能分支独立开发 |
| 答辩前系统不稳定 | 演示失败 | 提前准备固定数据、固定账号和演示脚本 |
| 功能太多来不及 | 核心流程不完整 | 优先保证登录、户籍、迁移、审批和查询可演示 |
