# Population_System 项目总结报告

## 一、项目概览

**Population_System** 是基于 **Spring Boot 3 + MyBatis-Plus** 的人口管理信息系统后端服务（artifactId: `population_miniterm`），为前端 Vue.js 应用提供 RESTful API，覆盖基层人口业务的完整场景。

- **运行环境**：JDK 17、Maven 3.8+、MySQL 8.0+、Redis 6.0+（可选）
- **服务端口**：`http://localhost:8080`，API 统一前缀 `/api`，统一响应 `Result<T>`
- **API 文档**：Knife4j 4.5.0，启动后访问 `/doc.html`

## 五、详细业务流程（核心业务链路）

### 5.1 整体业务主链路（来自 v4.0 设计）

```
business_application（业务申请）
    → application_material（申请材料，校验 VERIFIED）
    → sys_approval_request（审批主单，L3 必签）
    → sys_approval_log（审批日志）
    → 具体业务表（person / household / migration_in/out / cancellation 等）
    → residence_registration（当前户籍） + residence_archive（历史快照）
```

### 5.2 业务必交材料闸门（统一规则）

`ApplicationMaterialService#assertRequiredVerified(applicationId, businessType)` 是统一闸门，被 `PersonServiceImpl#createPerson`、`HouseholdServiceImpl#establishHousehold`、`ApprovalGateServiceImpl#approve` 在事务内调用：

| 业务类型 | AND 必交 | OR 备选组（任一即可） |
|---|---|---|
| `PERSON_REGISTER` / `PERSON_CREATE` | `IDENTITY_DOC` | — |
| `HOUSEHOLD_ESTABLISH` / `HOUSEHOLD_CREATE` | `IDENTITY_DOC` | `HOUSEHOLD_BOOKLET` / `RESIDENCE_PROOF` |
| `MIGRATION_IN_*` / `MIGRATION_OUT_*` | `IDENTITY_DOC`、`MIGRATION_CERT` | — |
| 其它业务 | 直接放行 | — |

校验要求：每份必交材料 `required_flag=1` 且 `verify_status=VERIFIED`。

### 5.3 标准调用顺序（L2 经办 → L3 审批 → 落库）

1. `POST /api/business-applications` → 拿到 `applicationId`
2. `POST /api/application-materials`（按必交清单上传材料）
3. `PUT /api/application-materials/{id}/verify?passed=true` → `verify_status=VERIFIED`
4. `POST /api/persons` 或 `POST /api/households/establish`，**必带** `applicationId`；闸门通过后落库（L3 直通 / L1·L2 走审批）
5. L3 调 `/api/approval-gate/approve/{approvalId}`，再次跑同一闸门，反序列化 payload，dispatch 到对应 Service 完成真实落库

### 5.4 迁出业务（核心事务边界，★★★★★）

```
@Transactional(rollbackFor = Exception.class)
MigrationOutServiceImpl.complete(outId, operatorId):
  1. SELECT migration_out FOR UPDATE（防重入）
  2. SELECT person, FOR UPDATE 锁当前居住登记
  3. 若存在：
     3.1 INSERT residence_archive（type=MIGRATION_OUT）
     3.2 DELETE residence_registration
     3.3 UPDATE household_member SET member_status='LEFT', leave_date=out_date
  4. UPDATE migration_out SET archive_id, operator_id, completed_at
```

**任一步骤失败全部回滚**，保证一人一条 + 当前与归档物理分离的不变式。

### 5.5 迁入业务（同市跨区/外来）

- 若 `CROSS_DISTRICT` 且已有当前登记：先按迁出流程归档 + 删除 + LEFT 成员，再 INSERT 新登记
- INSERT 新 `residence_registration`、`UPDATE person.record_status='ACTIVE'`、INSERT 新 `household_member (CURRENT)`、回填 `migration_in.new_registration_id`

### 5.6 人口注销 / 家庭户销户

- **前置校验**：未办结申请单数 = 0；销户额外要求无 CURRENT 成员
- 事务内走同一套 archiveAndRemove 流程，更新 `person.record_status='CANCELLED'` 和 `household.status='CANCELLED'`
- 抛 `PersonAlreadyHasRegistrationException`、`HouseholdNotEmptyException`、`HouseholdHasOutstandingApplicationException` 业务异常

### 5.7 立户流程（HOUSEHOLD_ESTABLISH）

- 校验必交材料闸门（IDENTITY_DOC + HOUSEHOLD_BOOKLET/RESIDENCE_PROOF 二选一）
- `FOR UPDATE` 锁户号去重
- 若 `head_person_id` 非空且户类型为 FAMILY，自动建 HEAD 成员关系（处理 `uk_member_current_dedup` 冲突时跳过）
- 触发 `IdentityMasker` 脱敏证件号

### 5.8 权限与三级审批模型

- `sys_role.permission_level` ∈ {1=L1_QUERY 查询级, 2=L2_HANDLE 经办级, 3=L3_APPROVE_ADMIN 审批/管理级}
- `data_scope_code` ∈ {ALL, DEPARTMENT, REGION, SELF}
- **JWT 签发时即写入** `permLevel/roleCode/dataScope/permCodes` 四元组 + Redis 缓存兜底
- `@RequiresPermission(value, all)`（RBAC 细粒度）+ `@RequiresLevel(N)`（粗粒度闸门）双切面
- Controller 内判断：`permissionLevel >= 3` 直通落库，否则走 `ApprovalGateService.submit` 生成审批单

### 5.9 一人一户一 CURRENT 去重（DB 层生成列）

```sql
current_dedup_key VARCHAR(40) GENERATED ALWAYS AS (
    CASE WHEN member_status='CURRENT'
         THEN CONCAT(household_id,'-',person_id)
         ELSE NULL END
) VIRTUAL,
UNIQUE KEY uk_member_current_dedup (current_dedup_key)
```

历史行（LEFT/CANCELLED）的 `current_dedup_key=NULL` 不受唯一键约束；换户主：旧行 LEFT + 新目标户 NEW。

### 5.10 当前户籍与历史归档（物理分离原则）

- `residence_registration` 一人一条（`uk_registration_person`），仅当前有效
- `residence_archive` 只 INSERT 不 UPDATE，存全字段快照
- 任何"当前登记消失"：**先 INSERT 归档快照，再 DELETE 当前登记**，事务保证原子性

## 七、统一响应 & 全局异常处理

### 7.1 `Result<T>` 统一响应

```json
{ "code": 200, "message": "ok", "data": { ... } }
```

- 分页场景 `data` 为 `PageVO<T>`（records / total / page / size）

### 7.2 `GlobalExceptionHandler` 覆盖

- `MethodArgumentNotValidException` / `BindException` / `ConstraintViolationException` → 400
- `BizException`（含 8 个子类）→ 按 `code` 返回（如 `NotFoundException`、`DuplicateException`、`IdCardInvalidException`、`PhoneInvalidException`、`HouseholdNotEmptyException`、`HouseholdHasOutstandingApplicationException`、`PersonAlreadyHasRegistrationException`、`ForbiddenException`）
- `DuplicateKeyException` → 409（翻译 `DuplicateException`）
- `DataIntegrityViolationException` → 400
- `RuntimeException` / `Exception` → 500

## 八、安全机制（JWT 三级权限）

1. **JwtAuthInterceptor**：除 `/api/auth/login` 外，所有 `/api/**` 请求必须携带 `Authorization: Bearer <token>`
2. **JWT Claims 必含字段**：`uid / uname / realName / permLevel / roleCode / dataScope / permCodes`；缺失任一字段视为旧 token 强制重登
3. **Redis 兜底**：`PermissionCache` 缓存用户权限码 TTL 与 JWT 一致；permCodes 为空时回退
4. **切面**：`PermissionAspect`（按 OR/AND 判定 RBAC 权限点）、`LevelGateAspect`（按 `permLevel` 阈值）
5. **业务接口**：每个 Controller 方法用 `@RequiresPermission(...)` 标注，部分关键操作（如 `ApprovalGateController`、`CancellationRecordController` 办结端点）用 `@RequiresLevel(3)`

## 十二、项目亮点

1. **完整的基层人口业务领域建模**：25 张表覆盖主档、户籍、迁移、归档、居住证、流动人口、重点人口、证件、业务审批等完整场景
2. **严格的当前/历史物理分离**：`residence_registration` vs `residence_archive` + 事务边界保证原子性
3. **业务必交材料闸门**：统一规则，被三种入口（直建/审批落地）调用，杜绝不合规业务数据落库
4. **三级权限 + RBAC + 数据范围**模型：JWT 携带完整四元组，AOP 双切面拦截
5. **审批联动机制**（`ApprovalGateService`）：L1/L2 提交草稿 → L3 真实落地业务 → 事务内反序列化 payload 派发，**不破坏现有 sys_approval_request 表结构**
6. **身份证 GB 11643-1999 校验**（加权因子 + 校验位 + 出生日期合法性）+ MySQL CHECK 双保险
7. **一人一户一 CURRENT**：DB 层 `GENERATED ALWAYS AS ... VIRTUAL` 生成列 + UNIQUE KEY 自动去重
8. **75 个索引覆盖高频查询**，核心外键默认 `RESTRICT` 保护关键数据
9. **统一响应 / 全局异常**：10+ 种异常类型映射 HTTP 状态码，敏感错误信息脱敏
10. **完整测试体系**：Mockito 单测覆盖核心 Service 事务边界，集成测试覆盖 JWT/Redis
