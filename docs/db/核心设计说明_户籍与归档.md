# 核心设计说明：当前户籍 vs 历史归档

> 这是 v4.0 最关键的边界规则。任何人口/户口的状态变更都必须满足本文档的"事务一致性原则"。

## 1. 物理分离原则

- **`residence_registration`** 仅保留当前有效的登记（一人一条，DDL 通过 `UNIQUE KEY uk_registration_person(person_id)` 保证）。
- **`residence_archive`** 是历史快照表，**只 INSERT 不 UPDATE**（除非修复错数据，但应保留日志）。
- 任何"当前登记消失"的动作，**必先**INSERT 归档快照，**再**DELETE 当前登记。

## 2. 事务一致性原则（★★★★★）

迁出/注销业务的实现必须满足以下顺序与回滚边界。**任何步骤失败，全部回滚**。

### 2.1 迁出（MIGRATION_OUT）

```
@Transactional(rollbackFor = Exception.class)
MigrationOutServiceImpl.complete(outId, operatorId):
  1. migration_out FOR UPDATE; 若 completed_at 非空 → BizException
  2. residence_registration FOR UPDATE by person_id; 允许不存在
  3. 若存在：
     3.1 INSERT residence_archive；得到 archive_id
     3.2 DELETE residence_registration by (person_id, registration_id)
     3.3 UPDATE household_member SET member_status='LEFT', leave_date=out_date
         WHERE person_id=? AND member_status='CURRENT'
  4. UPDATE migration_out SET archive_id=?, operator_id=?, completed_at=NOW()
```

**前置约束**：
- `migration_out.out_id` 在事务前已存在；
- 同批次同市跨区：`migration_in.complete` 与 `migration_out.complete` 互不阻塞（不同事务），由人工工作流保证顺序；
- 一人一条：`residence_registration.person_id` 在 COMMIT 之前是 locked 行，并发 INSERT 触发 `DuplicateKeyException` → 翻译为 `PersonAlreadyHasRegistrationException`。

### 2.2 迁入（MIGRATION_IN）

```
@Transactional(rollbackFor = Exception.class)
MigrationInServiceImpl.complete(inId, operatorId):
  1. SELECT migration_in
  2. SELECT person, SELECT target_household
  3. 若 in_type = CROSS_DISTRICT 且存在旧 registration：
     3.1 INSERT archive（MIGRATION_OUT）
     3.2 DELETE 旧 registration
     3.3 UPDATE household_member LEFT
  4. INSERT 新 residence_registration（person_id, household_id, MIGRATION_IN, in_date）
  5. UPDATE person.record_status_code = 'ACTIVE'
  6. INSERT household_member（OTHER, in_date, CURRENT）
  7. UPDATE migration_in SET new_registration_id, operator_id, completed_at
```

### 2.3 人口注销（PERSON_CANCEL）

```
@Transactional(rollbackFor = Exception.class)
CancellationRecordServiceImpl.completePersonCancellation(cancelId, operatorId):
  1. SELECT cancellation_record FOR UPDATE
  2. 前置：countOutstandingApplicationsByPerson = 0
  3. SELECT person；SELECT FOR UPDATE residence_registration
  4. 若存在：
     4.1 INSERT residence_archive（archive_type_code='PERSON_CANCEL'）
     4.2 DELETE registration
     4.3 UPDATE household_member SET member_status='CANCELLED', leave_date=cancel_date
  5. UPDATE person.record_status_code = 'CANCELLED'
  6. UPDATE cancellation_record SET archive_id, operator_id, completed_at
```

### 2.4 家庭户销户（HOUSEHOLD_CANCEL）

```
@Transactional(rollbackFor = Exception.class)
CancellationRecordServiceImpl.completeHouseholdCancellation(cancelId, operatorId):
  1. SELECT cancellation_record FOR UPDATE
  2. 前置：
       - countOutstandingApplicationsByHousehold = 0
       - countCurrentMembers = 0；否则抛 HouseholdNotEmptyException
  3. 写户级归档快照（借 head_person_id 的 registration 挂上 HOUSEHOLD_CANCEL 类型）
  4. DELETE head registration；UPDATE head 成员关系 CANCELLED
  5. UPDATE person（head）record_status_code = 'CANCELLED'
  6. UPDATE household.status = 'CANCELLED'
  7. UPDATE cancellation_record
```

## 3. 一人一户一 CURRENT 的去重约束

`household_member` 通过生成列 + 唯一键保证：

```sql
current_dedup_key VARCHAR(40) GENERATED ALWAYS AS (
    CASE WHEN member_status = 'CURRENT'
         THEN CONCAT(household_id, '-', person_id)
         ELSE NULL
    END
) VIRTUAL,
UNIQUE KEY uk_member_current_dedup (current_dedup_key)
```

- 历史行（LEFT/CANCELLED）的 `current_dedup_key = NULL` 不受唯一键约束。
- 同人同户的 CURRENT 行被锁约束。
- 换户主：`@Transactional` 下把旧行 LEFT，新目标户 NEW 一行 CURRENT（relationship=HEAD）。

## 4. CHECK 约束清单（v4.0 增强版）

| 表 | 约束 | 作用 |
|----|------|------|
| `person` | `chk_person_identity_no` | ID_CARD 时强校验 18/15 位 + REGEXP + 出生日期合法性（在 DTO/IdCardValidator 中校验，DDL 用 REGEXP 双保险） |
| `person` | `chk_person_phone` | 手机号必须 `^1[3-9]\d{9}$` |
| `person` | `chk_person_birth_date` | 出生日期不晚于今天 |
| `person` | `chk_person_deleted` | 软删除值 ∈ {0,1} |
| `admin_region` | `chk_enabled_flag` | enabled_flag ∈ {0,1} |
| `household_member` | `chk_leave_date` | leave_date 为空 OR ≥ join_date |
| `cancellation_record` | `chk_cancel_object` | PERSON 与 household_id 互斥；HOUSEHOLD 与 person_id 互斥 |
| `sys_approval_request` | `chk_required_level` | 1 ≤ required_level ≤ 3 |
| `sys_role` | `chk_permission_level` | 1 ≤ permission_level ≤ 3 |
| `residence_archive` | `chk_end_date_snapshot` | end_date_snapshot ≥ start_date_snapshot |
| `business_application` | `chk_completed_at` | completed_at ≥ submitted_at |

## 5. 重要异常与错误码

| 异常 | 触发场景 |
|------|---------|
| `PersonAlreadyHasRegistrationException` | 同一 person 重复当前登记（uk_registration_person） |
| `HouseholdNotEmptyException` | 销户时仍有 CURRENT 成员 |
| `HouseholdHasOutstandingApplicationException` | 未办结申请单关联 |
| `IdCardInvalidException` | GB 11643-1999 校验失败 |
| `PhoneInvalidException` | 手机号格式错误 |
| `DuplicateException` | DB 唯一键冲突（DuplicateKeyException 翻译） |
